# AGENTS.md

## Project overview

This repository contains `com.samsung.smart`, a Homey SDK v3 app written in TypeScript for controlling Samsung Smart TVs. It contains three protocol-specific drivers:

- [`drivers/Samsung/`](drivers/Samsung/) supports newer TVs through the Samsung WebSocket API on ports 8001/8002.
- [`drivers/SamsungEncrypted/`](drivers/SamsungEncrypted/) supports H/HU/J/JU/JS models that require encrypted or model-specific pairing.
- [`drivers/SamsungLegacy/`](drivers/SamsungLegacy/) supports older TVs through the legacy TCP protocol on port 55000.

## Driver maintenance policy

Only [`drivers/Samsung/`](drivers/Samsung/) is maintained. [`drivers/SamsungEncrypted/`](drivers/SamsungEncrypted/) and [`drivers/SamsungLegacy/`](drivers/SamsungLegacy/) are retained for existing users, but must not be changed unless the user explicitly requests it. The maintainer cannot reliably test those TV families.

- Do not refactor, reformat, modernize, or add features to the encrypted or legacy drivers as part of work on the maintained Samsung driver.
- Do not claim encrypted or legacy compatibility from unit tests alone.
- Prefer a maintained-driver override over changing shared behavior when the change is specific to modern TVs.
- Changes under `lib/` can still affect all three drivers. Inspect those effects, minimize them, and state any unverified compatibility risk.
- Do not remove the retained drivers from manifests, Flow filters, or packaging unless explicitly requested.
- Root Prettier excludes the encrypted and legacy driver directories to prevent incidental formatting changes.

## Runtime and tooling

- Use Node 24.16.0 as specified by `.nvmrc`. The project uses Homey SDK v3, TypeScript 6, npm, ESLint flat config, and Prettier 3.
- Install root dependencies with `npm ci`. The `@balmli/homey-samsung-encryption` package is resolved through GitHub Packages; configure authentication outside the repository when a clean local install needs it. Never commit a token.
- `cmd/` is a separate CLI package with its own package manifest, lockfile, TypeScript config, and lifecycle. Do not update it as a side effect of root app tooling or dependency work.
- Do not upgrade dependencies, change the Node baseline, migrate the test framework, or convert module systems as part of an unrelated change.

Useful root commands:

- `npm run build` — compile app sources to `.homeybuild/`.
- `npm run format` — format supported repository files with Prettier.
- `npm run format:check` — check formatting without changing files.
- `npm run lint` — run the Prettier check followed by ESLint.
- `npm run test:typecheck` — type-check application and test sources without emitting files.
- `npm test` — type-check, run the Mocha suite, run Homey validation through `posttest`, and restore Prettier formatting of generated `app.json`.
- `npm test --ignore-scripts` — run only the existing Mocha suite, as CI does before its separate Homey validation action.
- `homey app validate --level publish` — run publish-level manifest validation locally.
- `homey app run` — run the app on a configured Homey for integration testing.

## Repository map

- `app.ts` initializes app-level Homey Flow cards.
- `lib/SamsungBase.ts` defines the shared `SamsungClient` contract and common command behavior.
- `lib/BaseDevice.ts` owns shared Homey capabilities, polling, power transitions, migration, and device lifecycle behavior.
- `lib/BaseDriver.ts` contains common driver initialization.
- `lib/UPnPClient.ts` controls volume where TVs expose the UPnP service.
- `lib/SmartThings.ts` provides optional SmartThings input-source control.
- `lib/SamsungConfig.ts` and `lib/types.ts` define settings access and shared types.
- `drivers/Samsung/` contains the maintained modern client, Homey device, driver, pairing view, settings, and assets.
- `drivers/SamsungEncrypted/` and `drivers/SamsungLegacy/` contain retained, unmaintained protocol implementations.
- `.homeycompose/` is the source of truth for app metadata, Flow cards, and shared driver setting fragments.
- `app.json` is generated by Homey Compose. Do not edit it directly.
- `locales/en.json` contains user-visible translations and errors.
- `test/` contains the existing Mocha/Chai tests and lightweight Homey mocks. It is intentionally excluded from root Prettier until test modernization is requested.
- `cmd/` contains the separately managed command-line client.

## Protocol and device invariants

- Preserve the separation between Samsung remote control, UPnP volume control, Wake-on-LAN power-on, and SmartThings input control. They have different availability and failure modes.
- Keep wire formats, pairing, ports, and model-specific behavior in the relevant driver client. Put truly shared behavior in `lib/` only after checking its impact on retained drivers.
- When changing the `SamsungClient` interface, update every implementation required for compilation, but do not otherwise modify retained driver behavior without explicit approval.
- Preserve command validation: supported key codes, at most 100 expanded keys, and explicit delays between 1 and 9999 milliseconds.
- Preserve configurable key and channel delays; TVs may ignore commands sent too quickly.
- Preserve existing device data IDs, stored setting keys, capability IDs, and Flow card IDs. Existing paired devices and user Flows depend on them.
- Treat changes to settings, capabilities, pairing data, or store values as migrations. Migrations must be idempotent.
- Keep polling bounded and configurable. Avoid tight retry loops or broad LAN scans outside pairing.
- Pairing discovery must deduplicate devices and honor disconnect and timeout state.
- Clear Homey timers and stop rescheduling work after device deletion.

## Homey manifest and Flow changes

- Edit `.homeycompose/app.json`, `.homeycompose/flow/**/*.json`, driver `*.compose.json` files, or shared setting fragments as appropriate. Never hand-edit generated root `app.json`.
- Flow card IDs under `.homeycompose/flow/` must match registrations in `app.ts`.
- Driver filters must include only protocol families that implement the action or condition. Retained filters are not an invitation to change unmaintained drivers.
- Add or update `locales/en.json` for user-facing validation, availability, or pairing messages. Use `this.homey.__()` instead of hard-coded UI text.
- When adding or renaming a setting, update its compose definition, `DeviceSettings` in `lib/types.ts`, defaults, initialization, and `onSettings` handling for every explicitly supported driver.
- After compose changes, run publish-level Homey validation and inspect the generated `app.json` diff for unintended changes. Format `app.json` after generation.

## Coding conventions

- Prettier owns formatting for files not listed in `.prettierignore`. Run `npm run format` after editing supported files.
- Follow the existing architecture and keep changes narrowly scoped. Do not combine feature work with dependency upgrades, broad cleanup, or protocol refactoring.
- Use `DeviceSettings` enum values instead of repeating setting-key strings.
- Prefer existing interfaces and dependency injection (`config`, `logger`, `homeyIpUtil`, and device mocks) so logic remains testable.
- Use the app/device logger rather than `console.*`. Include relevant device and operation context without logging credentials.
- Await Homey state changes and network operations when ordering matters. Handle recoverable failures without terminating polling or lifecycle loops.
- Use `this.homey.setTimeout` and `this.homey.clearTimeout` in Homey-owned classes.
- Do not add synchronous blocking work to `onInit`, capability listeners, polling callbacks, or Flow listeners.
- Avoid new production dependencies when an existing dependency or small local implementation is sufficient.

## Security and privacy

- Never commit SmartThings tokens, Samsung pairing tokens, GitHub Packages tokens, device identifiers, MAC addresses, IP addresses, captured traffic, populated `env.json`, or diagnostic data identifying a user's home.
- Never log authentication tokens or complete credential-bearing URLs. Redact secrets in diagnostics and fixtures.
- Do not broaden TLS bypasses, network permissions, or remote access behavior without an explicit requirement and explanation.
- Tests must not contact a contributor's TV, Samsung account, Homey, or LAN by default. Mock network and Homey APIs for deterministic unit tests.

## Testing and verification

For setup, documentation, and code changes, run at least:

1. `npm run build`
2. `npm run lint`
3. `npm run test:typecheck`
4. `npm test --ignore-scripts`

Also run `homey app validate --level publish` for changes to manifests, drivers, capabilities, settings, locales, permissions, dependencies, or Flow cards. A full `npm test` performs the normal local test lifecycle when the Homey CLI is available.

- Add focused tests for changed pure behavior where practical, especially command parsing, delays, model classification, URI construction, and validation.
- Do not weaken timing, key-count, delay-range, model-routing, or input validation merely to make a test pass.
- Do not edit existing tests unless the task requires test work.
- Homey and TV hardware behavior cannot be established by compilation or mocks. State exactly which TV family and connection mode were tested on hardware and what remains unverified.
- Never represent the encrypted or legacy drivers as verified unless they were actually exercised on matching hardware.

## Issue and task workflow

- Handle one issue or task at a time. Finish its investigation, branch, implementation, verification, documentation, atomic commits, and pull request before starting another unrelated issue.
- Create a dedicated branch from the intended base branch. Use `fix/<issue-number>-<short-slug>` for bug fixes, `feat/<short-slug>` for features, and `chore/<short-slug>` for tooling or maintenance.
- Keep the branch and pull request limited to that one issue or coherent setup task.
- Use atomic commits. Each commit must contain one coherent change with directly required tests and documentation. Keep independently useful preparatory refactors, tooling changes, formatting, CI, and behavior fixes in separate commits.
- Commit only after the relevant focused checks pass. Do not hide unrelated cleanup inside a behavior commit.
- Prefer test-driven development for defects: first add or identify a focused regression test that fails for the expected reason, then make the smallest production change that passes it, and refactor only while keeping checks green.
- A locally testable defect should include regression coverage. Keep hardware-only verification requirements explicit when they cannot be automated.
- Push later atomic commits for the same task to the same branch and pull request. Do not mix another issue into that branch.
- In pull-request descriptions, include the issue or task, behavior or setup change, verification commands and results, generated-manifest impact, and hardware limitations.

## Release workflow

- Prepare a release only when the user explicitly requests a version. Start from an up-to-date default branch and create `release/<version>`.
- Treat `.homeycompose/app.json` as the source of truth for the Homey app version. Update its `version`, add the new version first in `.homeychangelog.json`, and add a matching newest-first section under `README.md` release notes.
- Do not synchronize the unrelated npm package version automatically. Change `package.json` version only when explicitly requested.
- Write user-facing release notes that summarize observable fixes, compatibility changes, and safety-relevant behavior. Do not claim unverified TV-family compatibility.
- Run `npm run format`, `npm run build`, `npm test`, and `npm run lint`. Confirm publish-level Homey validation succeeds, inspect generated `app.json`, and keep its version synchronized with `.homeycompose/app.json`.
- Review the complete diff for unintended generated-manifest or retained-driver changes, then create one atomic release commit.
- Push the release branch and open a pull request against the default branch when requested.
- Do not create a tag, GitHub release, publish to Homey, merge the pull request, or change release state unless the user explicitly requests that action.

## Completion checklist

- The change is scoped to one task and unrelated user changes are preserved.
- Only the maintained Samsung driver was changed unless retained-driver work was explicitly requested.
- Generated `app.json` was not edited directly and was inspected after validation.
- Flow metadata, translations, settings, capabilities, and runtime registrations remain synchronized.
- Build, formatting, lint, type-check, existing tests, and relevant Homey validation results are reported.
- Hardware-tested and unverified behavior are clearly distinguished.
- No credentials, local network details, generated build output, dependency directories, or reference symlinks were committed.
- Commits are atomic and release actions remain within the user's explicit authorization.
